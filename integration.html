Integration Guide

Welcome! This guide is designed for engineering teams integrating with the Nucleus Microservice. By integrating with this service, your application gains the ability to send Emails, SMS, and Real-Time In-App popups globally.

1. How It Works (The TL;DR)

The Notifications Microservice follows a strict philosophy: You provide the data; we handle the delivery.

  1. You trigger an event: When something happens in your app, your backend makes a simple HTTP POST request to our webhook.
  2. We compile the templates: We look at database templates tied to your project, merge your data into HTML/SMS/In-App content.
  3. We deliver it everywhere: We dispatch Email, SMS, and Real-time UI popup jobs using the tenant's configured providers.

2. Prerequisites & Onboarding

Before you can write any code, you need to be onboarded onto the Notifications Engine. Request the following from the Core Team:

  • Tenant API Key (BACKEND ONLY): Your privileged secret key used to trigger webhooks. Never expose this to a browser.
  • Tenant ID (FRONTEND IDENTIFIER): Your UUID identifier. Your frontend uses this ID when requesting realtime tokens.
  • Webhook Secret (OPTIONAL - HIGH SECURITY): An optional secret used to sign your webhook payloads with HMAC-SHA256.

3. Global Templates We Currently Support

The worker supports global templates by resolving any event type that starts with global..

eventType Typical Meaning Recommended Channel
global.infoInformational update, normal workflow progressPush, Email
global.successPositive completion or approval outcomePush, Email
global.warningReminder, upcoming deadline, or cautionary statusPush, Email
global.alertImportant issue, rejection, or urgent statePush, SMS

Recommended Payload Shape

{
  "eventType": "global.info",
  "payload": {
    "userId": "uuid-of-the-recipient",
    "recipientEmail": "user@example.com",
    "recipientPhone": "+12345678900",
    "title": "Service Request Created",
    "message": "Your request has been received and is under review."
  }
}

4. Triggering Events from Your Backend

Whenever a business event occurs that requires user notification, your server should make an HTTPS request.

API Endpoint: POST /api/v1/events/trigger

Headers:

Content-Type: application/json
x-api-key: YOUR_SECRET_TENANT_API_KEY
X-Nucleus-Signature: <your_computed_hmac_hex> # Optional High Security

Request Body:

{
  "eventType": "service.applied",
  "payload": {
    "userId": "uuid-of-the-recipient",
    "serviceName": "Premium Subscription",
    "recipientEmail": "user@example.com",
    "amount": "$49.99"
  }
}

Idempotency & Retries

To prevent duplicate notifications when your system retries a failed webhook, the engine derives an idempotency key from tenantId + eventType + payload. To explicitly override this, provide a unique key:

X-Idempotency-Key: your-unique-uuid-12345

5. Implementing Real-Time UI Popups (Frontend)

To display live pop-up notifications as soon as your backend fires an In-App event, your frontend needs to connect to the Centrifugo WebSocket Server.

Requesting a Real-Time Token

Get a secure JSON Web Token (JWT) from the Auth bridge.

POST /api/v1/auth/realtime-token
{
  "tenantId": "YOUR_TENANT_ID",
  "userId": "uuid-of-the-logged-in-user"
}

Connecting to the WebSocket Pipe

Pass the generated JWT into the Centrifuge SDK in your React/Vue app:

import { Centrifuge } from 'centrifuge';

const centrifuge = new Centrifuge('ws://notifications.yourdomain.com/connection/websocket', {
  token: response.token
});

response.channels.forEach((channelName) => {
  const sub = centrifuge.newSubscription(channelName);
  sub.on('publication', function (ctx) {
    showToast(ctx.data.title, ctx.data.body);
  });
  sub.subscribe();
});

centrifuge.connect();
