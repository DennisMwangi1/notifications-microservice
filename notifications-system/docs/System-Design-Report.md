# Notifications Microservice System Design Report

## 1. Project Purpose & Vision
The Notification Microservice is designed as a centralized, decoupled, omnichannel messaging engine. It acts as shared infrastructure for a broader ecosystem of internal projects (e.g., Fintech apps, E-commerce storefronts). 

Instead of embedding notification code, third-party provider SDKs, or WebSocket routing logic into individual applications, those external projects act strictly as **Event Producers**. They fire a generic webhook containing business data. The microservice then translates that raw event dynamically into highly formatted Emails, SMS, and Real-Time In-App Push notifications based purely on database configurations—completely abstracting the delivery complexity away from the client applications. 

It is built from the ground up on a **strict multi-tenant architecture**, ensuring zero data leakage, namespace collisions, or unauthorized channel access between different projects.

## 2. Architecture & Modules Implemented

### 2.1 Core Infrastructure
- **NestJS Worker (TypeScript/Node.js):** The "brain" of the operation. Handles REST APIs, Webhook ingestion, Template compilation (MJML & Handlebars), database interactions, and acts as the primary Kafka Consumer/Producer.
- **Go Gateway (Go):** A highly performant Kafka consumer dedicated entirely to external I/O execution and delivery. Consumes formatted payloads and pushes them to Centrifugo or simulates external Email/SMS providers.
- **Centrifugo:** The real-time WebSocket server. Handles concurrent WebSockets, JWT authentication, and pushes live server-to-client UI popups.
- **Kafka:** The asynchronous message broker. Decouples the webhook ingestion from heavy rendering logic, and rendering from external delivery.
- **PostgreSQL (Prisma ORM):** The persistent storage layer capturing audit logs, in-app notification history, multi-tenant definitions, and UI templates.

### 2.2 Implemented System Modules

#### Webhook Ingestion Module (`EventsController`)
- **Role:** The entry point for external projects.
- **Interaction:** Exposes `POST /api/v1/events/trigger`. Authenticates the tenant API Key, enriches the incoming JSON payload with internal mapping IDs (tenant UUID), and places it onto the generic `tenant.event.received` Kafka topic.

#### Auth & Security Module (`AuthController`)
- **Role:** Gatekeeper for Real-Time WebSockets.
- **Interaction:** Exposes `POST /api/v1/auth/realtime-token`. Dynamically queries PostgreSQL for the tenant's exact allowed channels and signs a secure JWT defining the user's explicit Centrifugo capability claims. This prevents users from subscribing to unauthorized topics.

#### Rendering Module (`RenderService`)
- **Role:** The design compiler for UI presentation.
- **Interaction:** Exposes `render` (Compiles responsive MJML XML into raw HTML for Emails) and `renderText` (Safely runs Handlebars to merge variables into Plain Text for SMS/Push subject lines).

#### Omnichannel Dispatch Module (`NotificationsController`)
- **Role:** The core Matrix Router.
- **Interaction:** Consumes `tenant.event.received` from Kafka. Fetches *all* templates (The Delivery Matrix) matching the specific Tenant + Event Type. It iterates through the array, branching into `EMAIL`, `SMS`, or `PUSH` execution paths. It updates persistent databases, formats strict routing instructions, and pushes the final dispatch objects to the `notification.dispatch` Kafka topic.

#### Persistence & History Module (`NotificationsController` REST APIs)
- **Role:** Solves the "Offline User" gap.
- **Interaction:** Exposes `GET` and `PUT` endpoints. Allows frontends to fetch unread notification history from the `in_app_notifications` table using strict `tenant_id` isolation boundaries, and marks them as `READ` once the user views them.

#### Gateway Delivery Module (`gateway/main.go`)
- **Role:** The edge dispatcher.
- **Interaction:** Consumes `notification.dispatch` from Kafka. Blindly executes based on the `ActionType` generated by the worker, invoking simulated Email/SMS API calls or triggering real live API calls to Centrifugo for WebSockets. Updates the SQL Audit Log upon completion.

## 3. How the Modules Interact (The Data Flow)

1. **Trigger:** The E-commerce backend hits `POST /api/v1/events/trigger` with `{ "eventType": "order.shipped", "payload": { "orderRef": "ABC" } }`.
2. **Ingest:** NestJS validates their API key, maps it to `tenantId: UUID-1`, and publishes the enriched payload to the Kafka topic `tenant.event.received`.
3. **Matrix Routing:** The NestJS Kafka Consumer picks it up. It queries the DB for *all* active templates where `tenant_id = UUID-1` AND `event_type = "order.shipped"`. It finds two active templates: a `PUSH` template and an `EMAIL` template.
4. **Execution: Push Branch:** Merges the payload into the text body. Inserts a row into the `in_app_notifications` table as `UNREAD`. Emits `ActionType: REALTIME` to Kafka `notification.dispatch` specifying the exact WebSocket channel (e.g., `ecommerce_store#user123`).
5. **Execution: Email Branch:** Merges the payload into MJML and compiles HTML. Inserts a row into `notification_logs` as `PENDING`. Emits `ActionType: EMAIL` to Kafka.
6. **Delivery:** The Go Gateway consumes both messages from Kafka. It pushes the `REALTIME` payload via HTTP to Centrifugo (which instantly beams it down the websocket pipe). It then "sends" the `EMAIL` via SMTP simulator and updates the audit log DB to `SENT`.
7. **Offline Sync:** If the user was offline and missed the websocket stream in Step 6, they log in an hour later. Their frontend queries `GET /api/v1/notifications/{tenantId}/{userId}` and retrieves their `UNREAD` sync payload from the PostgreSQL engine.

## 4. What This Microservice CAN Do Now
- **Generic Event Processing:** Accept arbitrary JSON business events from any authenticated tenant without requiring codebase redeployments to support new trigger types.
- **Omnichannel Orchestration:** Simultaneously blast Emails, SMS, and In-App Push popups from a single webhook trigger, differentiated natively via DB template configurations.
- **Tenant Isolation:** Strictly isolate data, templates, and real-time WebSocket capabilities natively enforced at the database level using `tenant_id` and strict Prisma `Where` constraints.
- **Real-Time WebSockets:** Push server-to-client UI updates over Centrifugo channels fully secured by server-signed JWT capabilities.
- **Offline History Synchronization:** Native "Notification Bell" support via REST endpoints, allowing users to catch up on missed websocket streams.
- **Audit Logging:** Maintain a persistent paper trail mapping delivery states for external providers (`PENDING` -> `SENT`).

---

## 5. What This Microservice CANNOT Do
- **Manage User Preferences:** The engine has absolutely no concept of user opt-ins/opt-outs. The external project (tenant) MUST query its own database (e.g., *"Did user unsubscribe from marketing emails?"*) **before** firing the webhook.
- **Send Actual Emails/SMS:** Currently, the Go Gateway relies on `time.Sleep()` simulations. The HTTP SDKs for SendGrid, AWS SES, or Twilio have not been functionally implemented inside the Go binary.
- **Handle Bounces/Provider Callbacks:** There is no webhook listener configured to accept SendGrid "bounced" or "dropped" events to dynamically update audit logs or suppress future emails.
- **Schedule Future Notifications:** The engine is purely reactive. It cannot parse commands like *"Send this notification in 3 days."*
- **Bulk Broadcasts without Loops:** Broadcasting to a million users currently requires a million webhook triggers. There is no batch processing or fan-out ingestion endpoint implemented.

---

## 6. Current Limitations & Technical Debt
1. **Raw SQL Dependencies:** The codebase utilizes raw Postgres commands (`$executeRaw`) for log inserts alongside Prisma ORM syntax. Migrating these to strict Prisma object notation (`prisma.notification_logs.create()`) would significantly improve typing and query maintainability.
2. **Synchronous Delivery Fallback:** If Centrifugo goes temporarily offline, the Go Gateway HTTP publish attempt will fail, output an error log, and aggressively discard the notification payload. A Dead-Letter Queue (DLQ) or retry loop native to the Go implementation is needed to guarantee delivery upon Centrifugo recovery.
3. **Go Gateway Monolith Coupling:** The Go Gateway currently relies directly on `lib/pq` to natively interact with Postgres to inject audit log updates. This violates the bounded microservice context. Ideally, the Gateway would emit a `notification.delivered` Kafka message back to NestJS, keeping all persistent DB operations restricted exclusively to the NodeJS worker.
4. **Lack of Automated Testing:** Despite high complexity covering the Omnichannel delivery matrix branches, there is no automated Jest Unit/Integration test coverage suite initialized.
