L2 UI Expansion Plan: Platform Operator Console and Tenant-Admin Console
Summary
Expand the frontend into two clearly separated surfaces inside the existing Next app:

the existing root surface remains the platform-operator console
a new /tenant/* surface becomes the tenant-admin console
This phase will deliver two complete journeys:

platform operators can provision a tenant, create the first tenant admin, generate a temporary password, and send the onboarding email
tenant admins can sign in, complete a forced first-login password reset, and manage their own tenant-scoped templates, template library, providers, logs, and DLQ items
The UI stays in one app for speed and shared components, but routing, auth state, and navigation must be fully separated so operator and tenant sessions do not bleed into each other.

Implementation Changes
1. Frontend Surface Split
Keep platform operator pages at the current root routes for this phase:
/login
/
/tenants
/logs
/dlq
/mail or /operational-mailer
Add a dedicated tenant route group:
/tenant/login
/tenant
/tenant/templates
/tenant/template-library
/tenant/providers
/tenant/logs
/tenant/dlq
/tenant/account
Create separate layouts and nav shells for platform and tenant users.
Do not reuse the current operator sidebar verbatim for tenants; tenant nav should be scoped to day-to-day tenant operations only.
Keep the existing template library and playground code intact for now, but do not make them central to the platform operator shell.
2. Auth and Session Model in the Shared App
Replace the current single-admin auth helper with actor-aware session handling.
Store platform and tenant sessions separately:
separate localStorage keys
separate route guards
separate logout flows
Platform routes must only accept platform_operator.
Tenant routes must only accept tenant_admin.
Add a tenant auth provider that reads the tenant token, validates expiry, and redirects unauthenticated users to /tenant/login.
Prevent cross-surface leakage:
platform token must not satisfy tenant routes
tenant token must not satisfy platform routes
Update UI identity display so tenant users see their tenant-scoped identity, not generic administrator text.
3. Platform Operator Provisioning and Onboarding
Expand tenant provisioning UI into a single flow that collects:
tenant details and limits
tenant delivery defaults
first tenant-admin username
first tenant-admin email
optional display name
After tenant creation, show a provisioning result card with:
tenant summary
first tenant-admin summary
onboarding delivery status
one-time temporary password
Add tenant detail actions to:
regenerate a temporary password
resend onboarding email
view onboarding delivery status
Replace the broken shared-integrations page with a platform Operational Mailer page used only for onboarding and other platform operational email.
Add a small platform-managed onboarding email template editor and preview surface, not a broad platform template suite.
4. Tenant-Admin Console
Build a tenant dashboard home page focused on tenant operations:
current tenant identity
provider count
template count
recent delivery activity
DLQ count
Build tenant pages against the existing tenant APIs:
templates: create, list, version history, deactivate
template-library: list, create, inspect entries
providers: create, list, rotate/update sender details
logs: paginated activity view with channel/status filters
dlq: list and retry failed notifications
Add a tenant account page for:
current admin identity
password change
first-login password reset completion
Keep tenant scope derived entirely from the tenant token. No tenant selector should exist anywhere in the tenant UI.
5. First-Login Password Reset
Complete the onboarding loop in this phase.
Extend tenant-admin login so a temporary-password user is still authenticated, but the UI immediately routes them into a required password reset step before they can use the console.
Block access to normal tenant pages until reset is completed.
After successful reset:
clear must_reset_password
stamp password change time
allow normal tenant navigation
Include a direct Change Password path on /tenant/account for later voluntary changes.
Public APIs, Interfaces, and Types
Extend tenant provisioning request/response to support first tenant-admin bootstrap:
request adds tenantAdmin: { username, email, displayName? }
response returns tenant, tenantAdmin, initialCredentials, and onboarding
Extend tenant_admins model/contracts to include:
email
display_name?
must_reset_password
password_set_at?
onboarding delivery status fields
Add tenant-admin credential lifecycle endpoints:
POST /api/v1/admin/tenants/:tenantId/admins/:adminId/reset-temporary-password
POST /api/v1/admin/tenants/:tenantId/admins/:adminId/resend-onboarding
Expand tenant auth responses so the frontend can enforce first-login flow:
login response should include mustResetPassword
GET /api/v1/tenant/auth/me should include username, email, displayName?, tenantId, and mustResetPassword
Add tenant password update endpoint:
POST /api/v1/tenant/auth/change-password
accepts current or temporary password plus new password
clears the forced-reset requirement when successful
Add platform operational mailer endpoints:
get/update platform mailer config
get/update onboarding email template
preview onboarding email render
Test Plan
Frontend auth tests:
platform and tenant sessions are stored separately
tenant routes reject platform-only sessions
platform routes reject tenant-only sessions
Provisioning tests:
platform operator can create tenant + first tenant admin in one flow
onboarding failure does not roll back tenant creation
resend/reset actions work and expose the temporary password only at creation/reset time
Tenant login tests:
temp-password login lands on forced reset
tenant cannot enter normal pages until reset succeeds
successful reset unlocks the console
Tenant console tests:
templates, library, providers, logs, and DLQ pages call only /api/v1/tenant/*
no tenant page allows arbitrary tenant IDs
provider pages never expose raw secrets after writes
Platform UI regression tests:
operator tenants page matches the new provisioning flow
operator mailer page replaces the broken shared-integrations behavior
existing platform ops pages still function
End-to-end scenario:
operator provisions tenant
tenant admin receives onboarding
tenant admin logs in with temp password
tenant admin resets password
tenant admin creates a provider and template
tenant admin can view logs and retry a DLQ entry within only their tenant scope
Assumptions and Defaults
The tenant-admin UI will live in the existing admin-ui app under /tenant/*, not in a new frontend repo.
Platform operator routes stay at the current root paths in this phase to avoid unnecessary migration churn.
Tenant-admin login remains username-based for now; email is required for onboarding and lifecycle management.
The first tenant-admin onboarding flow is complete in this milestone, including forced password reset.
The tenant console first release includes templates, template library, providers, logs, DLQ, and account/password management.
The current template library and playground implementation remains in the codebase and will be refined later rather than redesigned in this phase.
